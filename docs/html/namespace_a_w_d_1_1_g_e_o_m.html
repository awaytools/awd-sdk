Process Geometry.

Step 1: Validate the input.

We expect this input for each Geometry:

  • A list of Triangle.
    • A Triangle must contains a pointer for each Vertex. The Vertex must not be shared between Triangles.
    • A Triangle can have a FaceGroup-id. If no FaceGroup-id is provided, the Triangle will be assigned to the default FaceGroup.
  • A list of SharedVertices.
    Each SharedVertex contains a list of Vertex that share position.
    This are the Vertex that are contained in the Triangles
    • Each Vertex must have a position
    • It can also have other attributes like uv and vertex-normals, but also custom-attributes.
  • A list of meshes, that are using this Geometry.
    • Each Mesh has a map of FaceGroup-id / Material, that defines what material to use on the mesh for what Face-Group-id.
    • A Material that should be used as default material for this mesh, in case the mesh has no material mapped for a FaceGroup-id
    • A Material can have SubGeometrySettings, that define how to create the SubGeometry needed to display this mesh in regard to uv and vertex-normals.

If the list of triangles is emtpy, we return with error.

If the list of shared verticles is smaller than 3, we return with error.

The number of all Vertex found in SharedVertex must be three times the count of triangles. If not, we return with error.

If meshes are given, we check if they can be accessed as Mesh. If not, we return with error.

If no mesh is given, we still can create the Geometry, using the Settings, that are set on the geometry.

Step 2: Resolve Instances.

The Problem:

  • A Geometry can be used by multiple Meshes.
    SubGeometrySettings affect the output of the geomtry-proccessing.
    The SubGeometrySettings can be defined per Material, which are assigned to a Mesh.

    This follows, that we might need to create clones of the geometry in order to maintain all sub-mesh appearences.

The Solution:

  • Loop over all triangle, and for each mesh-instance, get the SubGeometrySettings to use, and compare it to the SubGeometrySettings of the other mesh-instances.
    This way, we calculate the minimum of Geometries needed to display all mesh-instances correctly.
    The geometries are than created, by cloning the original Geometry.
  • All Geometries are getting updated, so that contain the correct list of mesh-instances that should use them.
  • Mesh-instances also get updated to point to the correct Geometry, and the correct list of materials.

At this point we have a dedicated Geometry for each Mesh-instance that needs to be exported with own BlockSettings.

All coming steps will be executed once per Geometry.

Step 3: Prepare Geometry.

  • Get the data (mesh-list, shared_verts and triangles).
    This is either the original Geometry, or one of its clones, so the data already has been validated.
  • Loop over all Triangles, and create a SubGeom for each encountered SubGeo-id that is found on a Triangle.
    A SubGeom is used to fill a SubGeometry with data, without worrying about RessourceLimits.
    e.g. One SubGeom can hold multiple SubGeometries, in case Ressource-Limits was reached.
  • For each Vertex of a Triangle, apply the BlockSettings that should be used for this SubGeo as well as the facegroup_id and the subgeom_id of the triangle thats using this vert.
    This three things are needed to properly calculate the vertex-normals. We also set the set target-vertex and IDXmap to NULL for each Vertex.
    The Target-Vertex and ID-maps are later used to redirect a Vertex to another Vertex and its index in a SubGeometry-vert-list.
  • In this step we add the subgeocontrols to the geom, in synch with adding materials to mesh. If no material can be found for a mesh, we use default material.

Step 4: Calculate Vertex-Normals and prepare shared-vertex-lookup

  • We want calculate the vertex-data on Geometry-Level, before splitting into the actual SubGeometries. This way, we do we can preserve and smooth normals accross SubGeometries.
  • Loop over all SharedVertex:
    • Call calculate_normals() for each SharedVertex. This will calculate the normals based on the settings we set on all the verticles in previous step.
    • After calculating the normals, we call connect_duplicates() force each SharedVertex to check for duplicates inside its vert-list. When checking for duplicates, custom-attributes can be set to be considered when comparing vertices. If a duplicate is encountered, we set the target-vertex of this vertex to be the first encountered instance of this duplicates This will speed up look-up times when creating the final Vertex-lists for each subGeometry.

Step 5: Build SubGeometries.

  • Loop over triangles and fill previous created SubGeom with Triangles and vertices.
    For each Vertex of the Triangle, we check if it is a duplicate.
    If it is a Duplicate, we get the unique one (target_vertex()).
    Each uniqe Vertex stores a map with its index-position inside a subGeom with subGeom_id.
    This way we can ask a Vertex if it has already been exporter into a subgeom and it will report back the index.

Thats all.
The Geometries now have been processed completly, and should have a list of Subgeometries for each Geometry.
Each SubGeometry contains a list of Vertex and a list of Triangles.
The SubGeometry is now in a state, that it could be modifed (scale/merge).
The actual data-streams of a SubGeometry will never exist in libawd.
They only exist as recipe that can be written into a file.

Process Path Geometry.

Step 1: Validate the Geometry.

We expect this input for each Geometry:

  • A list of SubPathGeom.
    • A SubPathGeom has a type
      • NORMAL_FILL = a normal closed path that is filled with a fill-style, and can have holes
      • CONVERTED_FROM_STROKS = a normal closed path that is filled with a fill-style and was generated froma stroke, and can have holes
      • FILL_FOR_FONT = a set of pathes that are generated for a gylph in a font
    • A SubPathGeom has a multiple pathes (list of PathSegment)
      • A PathSegment contains min 2 and max 2 Vertex2D
    • A material that is asociated with this Shape. can be empty (fonts)

All coming steps are performed for each SubPathGeom found in the Geometry
Maybe we will even need to check intersections across SubPathGeom (?)

Step 2: Validate and prepare the SubPathGeom.

  • For each path inside a SubPathGeom, we do this:
  • We check if the path contains less than 3 PathSegmentssegments, and if so, we add a warning-message to the geometry.
  • check if the path is closing, and otherwise add a warning-message to the geometry.
  • If all path of this SubPathGeom are invalid, we will skip to next SubPathGeom, and add a warning message.
  • if this is coming from font, than we we need to decide what path is a hole, and what path is a contour.
    we store the results in a list of bools for later use.

Step 3: Calculate Curve-Types.

  • Up until here, curves have only been handled as 2 types. OUTTER_EDGE and CURVED_EDGE.
  • Before tesselation, we want to know the exact curve-types (convex / concave).
    We want to know this, because in tesselation, we only want to include the control-points for concave curves.
    For convex-curves and straight lines, we only want to include strat-point and end-point in tesselation.

    Problem:
  • For each path and each curved-segment, we need to decide the type of curve.

    Solution:
  • We can solve this testing if the control-point of the curve is outside or inside the path that contains this segment.
    If it is inside, and the path is a contour, than the curve will be convex.
    If it is outside, and the path is a contour, than it is concave.
    If the path is a hole, its the otherway around.
  • This solution will fail if the path self-intersects or gets very close to itself.
    To fix this, instead of testing if the control-point is contained in the path,
    we calculate the center of the line start-point <-> end-point, and move it 1 pixel in direction of the control-point.
    Now even for nearly-intersecting pathes, we can savly decide what type of curve this is.
  • A nice side-effect of this work-around is, that when testing for both the control-point, and the calculated-point,
    than we can compare the results, and if they differ,we know the curve is intersecting with the path , and can solve this by subdividing the segment.
  • If we want to resolve intersections, we repeat testing the above steps until either we dont find any more intersections, or the limit of iteration is reached.

For each PathSegment inside the path, we do this:

  • If the PathSegment is edge-type: OUTTER_EDGE, we do nothing.
  • If the PathSegment is edge-type: CURVED_EDGE, we try to figure out if it is concave or convex.

Optionally we test if the curve is intersecting with the rest of the path, and subdivide it, if it does.

  • To set the curve_type based on if the control-point of the curve is inside the path or not,
    we need to consider if this is a contour-path, or a hole-path.

In case the SubPathGeom was generated for a font, we already figured out if the path is a contour or a hole

In case the SubPathGeom is coming from a normal fill or from a fill converted from stroke, only the first path of a SubPathGeom should be considered a contour. all other should be considered a hole.

In case this path should be considered a hole, we invert the result coming from the "is inside path" function.

Now we can set the edge-type on the PathSegment

Step 4: Optionaly Resolve Intersections

  • At this point we now the type of curves for all curves.
    We also might have reolved some of the interecting curves of a path, by subdividing them.
  • Before tesselation, we want to resolve all intersecting curves in all pathes inside a SubPathGeom, by subdividing them.


Problem:

  • Find intersecting curves, and decide if the intersection can be resolved by subdividing one or both of the intersecting curves.

    Solution:
  • TBD

Step 5: tesselation.

  • Create a new tesselator from libtess2.
  • For each path, create a list of Tessreal.
    Loop over list of PathSegments, and create a TESSreal for the StartPoint. If the PathSegment is a convex curve, we also add the control-point of this PathSegment. Each list of TESSreal is added as contour to the tesselator.
  • Call the tesselation function on the tesselator. If the tesselation-function is not exectuted without error, we add a warning-message and skip to next SubPathGeom

Step 6: Create and fill SubGeometries.

  • For each SubPathGeom, we create a new SubGeometry.
    The SubGeometries can optionally be merged later.
  • we want the triangle-list to be in synch with the path-order. e.g. we run over the path and for each segment we add 1 triangle to the list. triangles that are considered to be inner-trianlges (not having any outside edges) are coming last.
